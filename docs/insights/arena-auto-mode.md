# Arena Auto Mode -- 产品思考文档

## 解决什么用户问题

### 核心痛点：AI Agent 表现难以系统化评估

当用户为特定场景（客服、健康管家、销售等）编写 prompt 时，他们面临的核心挑战是：**无法系统化地验证 prompt 在各种边界条件下的表现**。

目前的评估方式：
- 手动输入几条测试问题，凭感觉判断好不好
- 调一版 prompt → 手动测几轮 → 再调 → 再测，循环低效
- 没有结构化的评估标准，不知道"好"的定义是什么
- 无法追踪迭代过程中的进步/退步

Arena 的解决方案：**用 AI 对 AI 的自动化闯关测试，替代手动的随机测试**。

- Gatekeeper（守门人）根据场景描述自动生成提问，模拟真实用户行为
- Challenger（挑战者）执行被测 prompt，回答提问
- Grader（评分官）按预定义的结构化标准（必须项 Pass/Fail + 表现项 A/B/C/D）评判
- 关卡配置 JSON 文件定义了测试场景和评分标准，可版本控制、可复用

### Auto Mode 的价值

Auto Mode（自动模式）是 Arena 的核心交互形式。用户只需选择关卡、点击"开始挑战"，然后观看 Gatekeeper 与 Challenger 的自动对话，最后看到评分报告。整个过程无需人工干预。

这使得：
- 一次点击完成一整轮测试（5-15 轮对话 + 评分）
- 关注结果而非过程 -- 评分报告直接告诉你哪里好、哪里差
- 可重复执行 -- 同一关卡跑多次，观察 prompt 调整的效果

## 关键设计决策

### 1. 为什么选择单页状态切换而非多页路由

**决策：** Arena 页面内部通过 React state 切换三个阶段（关卡列表 → 运行对话 → 评分报告），不使用 Next.js 路由跳转。

**理由：**
- **状态是临时的。** 运行过程是一次性的，刷新页面后回到关卡列表是合理的。不需要 URL 能回到"某次运行的第 7 轮"。
- **切换更流畅。** 状态切换是即时的，无路由跳转开销。对话流到评分报告的过渡是在同一滚动容器内完成的，体验连贯。
- **简化架构。** 不需要管理 /arena/run/:id 等动态路由，减少复杂度。
- **与现有模式一致。** CodePilot 的聊天页面也是在同一视图内切换会话，不为每个会话创建 URL。

**权衡：** 牺牲了 URL 可分享性（无法通过 URL 直接跳转到某次运行）。但 v1 没有多用户协作场景，这不是问题。

### 2. 为什么 SSE 而非 WebSocket

**决策：** 使用 Server-Sent Events (SSE) 传输对话流，而非 WebSocket。

**理由：**
- **单向即可。** Arena 对话流是服务端 → 客户端的单向推送。用户发起运行后，不需要在运行过程中向服务端发送消息（取消通过 AbortController 中断连接实现）。
- **HTTP 语义更自然。** SSE 基于 HTTP 长连接，与 Next.js API Routes 天然兼容。WebSocket 需要额外的升级协议和连接管理。
- **项目已有 SSE 模式。** CodePilot 的聊天流式输出已使用 ReadableStream + TextEncoder 模式，Arena 直接复用这一基础设施。
- **重连更简单。** SSE 的 EventSource 内置自动重连；虽然 Arena 使用 fetch + ReadableStream 实现（非原生 EventSource），但简单的单次连接模型已足够。
- **无需双向通信状态管理。** WebSocket 需要心跳、重连、连接池管理等额外复杂度，对 Arena 的场景来说是过度设计。

**权衡：** 如果未来支持"人工 Gatekeeper 模式"（用户实时输入提问），可能需要重新评估。但该模式下可以用独立的 POST 请求发送用户消息，仍不需要 WebSocket。

### 3. 三角色模型的设计理由

**决策：** Arena 使用 Gatekeeper - Challenger - Grader 三角色分离模型。

**为什么不是二角色（对话 + 评分）：**
- 将"提问者"和"评分者"分离，避免评分偏差。如果提问者自己评分，它可能倾向于认为自己的提问引导得很好。
- Gatekeeper 关注"如何模拟真实用户场景"，Grader 关注"如何客观评判回答质量"，两者的 prompt 目标不同。

**为什么 Challenger 不直接对接外部 Agent：**
- v1 的目标是验证核心流程。Challenger 使用 Vercel AI SDK + System Prompt 内部模拟，已足够测试 prompt 质量。
- 对接外部 API（如已部署的 Agent 服务）需要处理认证、超时、格式适配等额外复杂度，留到后续版本。

**为什么 Grader 用离散评分而非数值：**
- LLM 在精确数值打分（如 1-100 分）上的一致性很差，同样的回答可能在两次评估中得到 72 分和 85 分。
- 离散判断（Pass/Fail、A/B/C/D）更稳定，LLM 更擅长分类任务。
- 对用户来说，"必须项是否通过 + 表现等级"比"总分 78.5"更有可操作性。

### 4. 气泡聊天布局的选择

**决策：** Gatekeeper 消息靠左、Challenger 消息靠右，类似即时通讯的对话布局。

**理由：**
- 用户对聊天气泡布局的心智模型已经非常成熟（微信、iMessage 等）
- 与 CodePilot 现有的聊天界面风格一致，降低认知负担
- 左右对齐 + 角色颜色（蓝/橙）+ 图标（盾牌/闪电），三重视觉线索确保角色辨识清晰

### 5. 评分报告在对话流下方展开

**决策：** GradeReport 渲染在 ConversationStream 的最后一条消息下方，在同一个滚动容器内。

**备选方案被放弃的原因：**
- **弹窗/模态框：** 遮挡对话内容，用户无法同时看到对话和评分
- **跳转到独立页面：** 失去对话上下文，用户需要在两个页面间切换
- **侧边面板：** 在桌面端可行但移动端体验差，且压缩了对话区域

**当前方案的优势：**
- 用户可以向上滚动回顾对话，对照评分中的"轮次 N"引用
- 符合时间线的自然阅读顺序：对话结束 → 评分出现
- 评分报告的入场动画（从下方滑入）给用户一个明确的"结果来了"的信号

## 竞品参考

### ChatGPT Arena / LMSYS Chatbot Arena
- 侧重于模型之间的横向比较（A/B 盲测）
- Arena 侧重于同一 Agent 在预定义场景下的深度测试
- 两者目标不同：Chatbot Arena 回答"哪个模型更好"，CodePilot Arena 回答"我的 prompt 在这个场景下表现如何"

### PromptFoo
- 优秀的 CLI 工具，批量评估 prompt 变体
- 但是面向开发者，缺乏直观的可视化交互
- Arena 提供了更直观的"观战"体验，适合非技术背景的 prompt 设计者

### Langchain Evaluation
- 框架级的评估工具链
- 需要编写 Python 代码配置评估流程
- Arena 通过 JSON 关卡配置 + GUI 操作降低了使用门槛

## 未来方向

### 短期（Phase 3）
- **世界分组 + 顺序解锁：** 关卡按世界组织，通关前一关解锁下一关，形成渐进式测试路径
- **历史记录：** 查看过往运行的对话和评分，追踪 prompt 迭代效果
- **人工 Gatekeeper 模式：** 用户手动输入提问，替代 AI 自动提问

### 中期
- **批量运行：** 一键运行同一关卡 N 次，统计通过率和评分分布
- **关卡编辑器：** 可视化创建和编辑关卡配置，替代手动编辑 JSON
- **评分对比：** 两次运行的评分并排对比，高亮差异

### 长期
- **外部 Agent 对接：** Challenger 角色调用外部已部署的 Agent API，测试真实线上服务
- **示范回放：** 展示某次优秀运行的完整对话过程，作为 prompt 调优的参考
- **自定义 Grader prompt：** 用户编写自己的评分标准 prompt，支持领域特定的评估维度

## 已知局限

1. **评分一致性：** 虽然离散评分比数值评分更稳定，但 LLM 评分仍有随机性。同一个 Agent 跑同一关卡两次，评分可能不完全一致。缓解方案：批量运行取均值（中期目标）。

2. **Gatekeeper 质量：** AI Gatekeeper 的提问质量依赖场景描述 prompt 的质量。如果场景描述模糊，Gatekeeper 可能提出过于宽泛或不够有挑战性的问题。

3. **单次运行视角：** v1 只展示最近一次运行的对话和评分，无法比较多次运行的趋势。历史记录功能将在 Phase 3 补充。

4. **关卡配置门槛：** 需要手动编写 JSON 文件定义关卡，对非技术用户不够友好。关卡编辑器将在中期迭代中加入。
